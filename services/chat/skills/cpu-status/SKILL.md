# cpu-status — CPU Fallback Status

## Automatic CPU fallback detection

When you receive a response whose metadata or system context indicates
`X-Fallback-CPU: true` or `fallback: true`, **before presenting the response**, display
this banner at the top of your response:

```
⚡ **CPU Response — reduced model** (`qwen3.5:0.8b`)
The GPU is currently in use. This response is generated by the backup CPU model.
Type `/wait-gpu` to cancel and wait for a GPU slot (higher quality response).
---
```

Then present the response normally below this banner.

## Slash command: /wait-gpu

When the user types `/wait-gpu`:

### 1. Confirm the cancellation

Respond immediately:
```
⏳ Canceling the CPU stream...
I'm putting your request on hold waiting for a GPU slot.
I'll notify you as soon as a GPU model is available.
```

### 2. Call the proxy to cancel and wait

```bash
curl -sf -X POST http://localhost:11435/cancel-and-wait \
  -H "Content-Type: application/json" \
  -d '{"requestId":"'"${LAST_REQUEST_ID:-unknown}"'","messages":[],"waitForGpu":true}' \
  --no-buffer 2>/dev/null &
WAIT_PID=$!
echo "Waiting for GPU (PID=$WAIT_PID)..."
```

### 3. Wait for GPU notification

Poll the scheduler status every 10 seconds and inform the user:

```bash
PROXY_URL="${PROXY_URL:-http://localhost:11435}"
SCHEDULER_URL="${SCHEDULER_URL:-http://openclaw-agent:7070}"
MAX_WAIT=300  # 5 minutes max

for i in $(seq 1 $((MAX_WAIT/10))); do
  sleep 10
  STATUS=$(curl -sf "${SCHEDULER_URL}/status" 2>/dev/null)
  FREE=$(echo "$STATUS" | node -e \
    "const d=require('fs').readFileSync('/dev/stdin','utf8'); \
     try{const j=JSON.parse(d); process.stdout.write(String(j.totalFree||0));}catch{process.stdout.write('?');}" \
    2>/dev/null)

  if [ "$FREE" != "0" ] && [ "$FREE" != "?" ]; then
    echo "✅ GPU slot available (${FREE}GB VRAM free) — you can resend your question."
    exit 0
  fi
  echo "⏳ Still waiting... (${i}0s elapsed, free VRAM: ${FREE}GB)"
done

echo "⚠️ GPU wait timeout exceeded (5min). You can retry now or continue on CPU."
```

### 4. Notify the user

When a GPU slot is available, display:
```
✅ **GPU slot available!**
You can now resend your question — it will be processed by the full GPU model.
```

## Model information

If the user asks which model is being used:

```bash
curl -sf "${SCHEDULER_URL:-http://openclaw-agent:7070}/status" | \
  node -e "
    const d = require('fs').readFileSync('/dev/stdin','utf8');
    try {
      const j = JSON.parse(d);
      const loaded = j.loadedModels || [];
      const chat   = j.chatSlots   || [];
      console.log('=== GPU Status ===');
      console.log('Free VRAM:', j.totalFree + '/' + j.totalVram + 'GB');
      console.log('Mode:', j.mode);
      console.log('Active surface:', j.activeSurface || 'none');
      if (loaded.length) {
        console.log('Loaded models:');
        loaded.forEach(m => console.log(' -', m.modelId, m.vram+'GB', m.agents+'/'+m.maxAgents, 'agents'));
      }
    } catch(e) { console.log('Scheduler unavailable'); }
  " 2>/dev/null || echo "Scheduler unavailable"
```

Present this information in a readable format to the user.

## Slash command: /gpu-status

When the user types `/gpu-status`, execute the command above and display the full
scheduler status in a formatted way.

## Important rules

- Only display the CPU banner when the fallback is actually active — not on every message
- The `/wait-gpu` slash command only makes sense if a CPU stream is in progress or has just finished
- Never automatically resend the request without explicit confirmation from the user
- If the scheduler is unavailable, inform the user that the GPU status is not accessible
