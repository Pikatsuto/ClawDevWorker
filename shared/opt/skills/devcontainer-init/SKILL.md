# devcontainer-init — Generate project devcontainer

## Commands

| Command | Action |
|---------|--------|
| `/devcontainer init` | Generate .devcontainer/ from BMAD artifacts or project analysis |
| `/devcontainer init --stack node,python` | Force specific stacks |

## When to use

- Automatically after BMAD completes in `/spec init` (the chat orchestrator calls this)
- Manually when a project has no `.devcontainer/` yet
- When the project stack changes and the devcontainer needs updating

## /devcontainer init Procedure

### 1. Detect the project stack

Read BMAD artifacts if available:
```bash
# Check for BMAD architecture document
for dir in _bmad-output docs/bmad .bmad bmad; do
  [ -f "$WORKSPACE/$dir/planning-artifacts/ARCHITECTURE.md" ] && ARCH_FILE="$WORKSPACE/$dir/planning-artifacts/ARCHITECTURE.md" && break
done

# Check for PRD
for dir in _bmad-output docs/bmad .bmad bmad; do
  [ -f "$WORKSPACE/$dir/planning-artifacts/PRD.md" ] && PRD_FILE="$WORKSPACE/$dir/planning-artifacts/PRD.md" && break
done
```

If no BMAD artifacts, analyze existing code:
```bash
# Detect stack from existing files
[ -f "$WORKSPACE/package.json" ] && STACKS="$STACKS node"
[ -f "$WORKSPACE/requirements.txt" ] || [ -f "$WORKSPACE/pyproject.toml" ] && STACKS="$STACKS python"
[ -f "$WORKSPACE/Cargo.toml" ] && STACKS="$STACKS rust"
[ -f "$WORKSPACE/go.mod" ] && STACKS="$STACKS go"
[ -f "$WORKSPACE/Gemfile" ] && STACKS="$STACKS ruby"
```

### 2. Generate devcontainer.json

Based on detected stacks, generate `.devcontainer/devcontainer.json`:

- `image` or `build.dockerfile`: choose the right base (node, python, rust, go, or multi-stack)
- `features`: add devcontainer features for detected tools (git, docker-in-docker, etc.)
- `postCreateCommand`: install project dependencies (`npm install`, `pip install -r requirements.txt`, etc.)
- `customizations.vscode.extensions`: recommend extensions for the detected stack
- `forwardPorts`: common ports for the stack (3000 for node, 8000 for python, etc.)
- `mounts`: mount `project_data` volume for OpenClaw shared state

### 3. Generate Dockerfile (if multi-stack or custom needs)

If the project uses multiple stacks or needs custom setup, generate a `.devcontainer/Dockerfile`:

```dockerfile
# Auto-generated by /devcontainer init
FROM mcr.microsoft.com/devcontainers/base:alpine

# Stack: [detected stacks]
RUN apk add --no-cache [packages for detected stacks]

# Project dependencies
COPY . /workspace
RUN [install commands for each stack]
```

### 4. Commit

```bash
git add .devcontainer/
git commit -m "feat: add devcontainer configuration for [stacks]"
```

## Stack templates

| Stack | Base image | Key packages | Extensions |
|-------|-----------|--------------|------------|
| Node | `node:24-alpine` | `npm`, `yarn`, `pnpm` | `dbaeumer.vscode-eslint`, `esbenp.prettier-vscode` |
| Python | `python:3.12-alpine` | `pip`, `virtualenv` | `ms-python.python`, `ms-python.vscode-pylance` |
| Rust | `rust:alpine` | `cargo`, `rustfmt`, `clippy` | `rust-lang.rust-analyzer` |
| Go | `golang:alpine` | `go` | `golang.go` |
| Multi | `alpine:3.21` | all needed packages | all relevant extensions |

## Integration with /spec init

After BMAD generates the architecture document, `/spec init` calls this skill automatically:
1. BMAD completes → architecture.md contains the tech stack
2. This skill reads architecture.md → detects stacks
3. Generates `.devcontainer/devcontainer.json` (+ Dockerfile if needed)
4. Files are committed alongside the BMAD artifacts
5. When the user runs `/dev create`, the devcontainer is ready
